# API 代理转发工具 Chrome 插件

## 功能介绍

这是一个功能强大的 Chrome 浏览器插件，允许你拦截网页的 API 请求，并将它们转发到不同的服务器。主要用于测试环境到生产环境的请求代理。

### 核心功能

1. **多规则转发** - 支持配置多个转发规则，每个规则可独立启用/禁用
2. **灵活的 URL 匹配** - 使用正则表达式匹配源 URL，支持复杂的路由匹配
3. **自动 Cookie 管理** - 自动获取目标服务器的 Cookie，也支持手动配置
4. **自定义 Headers** - 支持为转发请求添加自定义 HTTP Headers
5. **完整的日志系统** - 记录所有请求和响应，支持搜索、过滤和导出
6. **数据导入导出** - 支持备份和导入规则配置

## 安装方式

### 方法1：手动加载（开发者模式）

1. 打开 Chrome 浏览器，访问 `chrome://extensions/`
2. 启用右上角的"开发者模式"
3. 点击"加载未封装的扩展程序"
4. 选择项目文件夹 `apiProxy`
5. 插件将被加载到你的浏览器中

### 方法2：使用已封装的 CRX 文件

1. 如果已有 `.crx` 文件，可直接拖到 `chrome://extensions/` 页面安装

## 快速开始

### 1. 创建第一个转发规则

1. 点击浏览器工具栏中的插件图标
2. 点击"+ 添加新规则"按钮
3. 填写以下信息：
   - **规则名称**：例如"测试环境到生产环境"
   - **源 URL 模式**：例如 `https://test\.example\.com/(.*)`
   - **目标 URL**：例如 `https://api.example.com/$1`
   - **请求方法**：选择 ALL 表示所有方法
4. 点击"保存规则"

### 2. 配置 Cookies

如果需要 Cookie 认证：

1. 编辑规则
2. 打开 Cookie 设置
3. 选择"自动获取 Cookie"，插件会自动从目标 URL 获取
4. 也可以点击"手动配置 Cookie"添加特定的 Cookie 值

### 3. 添加自定义 Headers

某些 API 可能需要额外的 Headers：

1. 编辑规则
2. 在"自定义 Headers"部分点击"+ 添加 Header"
3. 输入 Header 名称和值

### 4. 查看日志

1. 切换到"日志"标签页
2. 实时查看所有拦截和转发的请求
3. 使用搜索框或过滤器查找特定请求
4. 点击日志项查看详细信息

## 使用场景

### 场景1：测试环境到生产环境

假设你有两个环境：
- 测试环境：`https://test.app.com`
- 生产环境：`https://api.prod.com`

配置规则：
- 源 URL 模式：`https://test\.app\.com/(.*)`
- 目标 URL：`https://api.prod.com/$1`

这样测试环境的所有请求都会被转发到生产环境。

### 场景2：本地开发服务器

- 源 URL 模式：`https://api\.example\.com/(.*)`
- 目标 URL：`http://localhost:3000/$1`

可以在本地快速测试生产 API。

### 场景3：跨域请求

用于绕过某些跨域限制，通过转发到中间服务器实现代理。

## 正则表达式语法

源 URL 模式使用标准正则表达式，目标 URL 中可使用捕获组：

- `$0` 或 `$&` - 完整匹配
- `$1, $2, ...` - 第 1, 2, ... 个捕获组

### 例子

```
源模式：https://test\.example\.com/api/(v\d+)/(.*)?id=(\d+)
目标：https://prod.example.com/$1/$2?id=$3

请求：https://test.example.com/api/v2/users?id=123
转发到：https://prod.example.com/v2/users?id=123
```

## 常见问题

### Q: Cookie 自动获取不到怎么办？

A: 
1. 确保已在另一个标签页登录到目标服务器
2. 可以手动配置 Cookie，复制目标服务器的 Cookie 值
3. 点击"刷新 Cookie"按钮重新尝试获取

### Q: 某些请求转发失败怎么办？

A:
1. 检查日志面板查看失败原因
2. 确认源 URL 模式是否正确匹配
3. 检查是否需要添加特定的 Headers（如 Authorization）
4. 确认目标服务器是否正常运行

### Q: 如何导出规则配置？

A:
1. 切换到"规则"标签页
2. 点击"导出规则"按钮
3. 保存为 JSON 文件

### Q: 如何恢复已保存的规则？

A:
1. 切换到"规则"标签页
2. 点击"导入规则"按钮
3. 选择之前导出的 JSON 文件

## 技术架构

### 核心组件

- **manifest.json** - 扩展配置文件
- **background.js** - Service Worker，处理请求转发和规则管理
- **content-script.js** - 内容脚本，连接页面和后台
- **injected.js** - 注入到页面的脚本，拦截 fetch 和 XHR
- **popup/** - 插件界面（HTML/CSS/JavaScript）
- **managers/** - 规则、Cookie、日志管理模块

### 数据流

```
网页请求
  ↓
injected.js（拦截 Fetch/XHR）
  ↓
content-script.js（转发消息）
  ↓
background.js（匹配规则、获取 Cookie、执行转发）
  ↓
目标服务器
  ↓
响应返回到网页
```

## 权限说明

该插件需要以下权限：

- `storage` - 存储规则和日志配置
- `cookies` - 访问和管理 Cookies
- `tabs` - 获取当前标签页信息
- `scripting` - 注入脚本到网页
- `<all_urls>` - 拦截所有 URL 的请求

## 隐私和安全

- 所有数据（规则、日志、Cookies）都保存在本地浏览器中
- 不会上传任何数据到远程服务器
- 可随时清空所有日志和配置

## 故障排除

### 插件未加载

1. 确认 manifest.json 文件存在
2. 检查 Chrome 版本是否支持 Manifest V3（v88+）
3. 在控制台查看错误信息

### 请求未被拦截

1. 确认规则已启用（图标应显示为绿色）
2. 检查源 URL 模式是否正确
3. 在日志面板查看是否有错误信息

### Cookie 未生效

1. 确认 Cookie 已启用
2. 检查目标服务器是否正确返回了 Cookie
3. 尝试手动添加 Cookie

## 开发和扩展

### 修改源代码

1. 编辑相关 JavaScript 文件
2. 无需重新打包，刷新插件即可生效
3. 在 `chrome://extensions/` 中点击"刷新"按钮

### 调试

1. 打开 popup 右键选择"检查"打开开发者工具
2. 查看控制台输出和 Network 面板
3. 查看后台 Service Worker：在 `chrome://extensions/` 中找到插件，点击"Service Worker"
